# 效能分析

> 读取各功能的 summary 打点文件，生成结构化的效能分析报告。
> 本技能不修改任何工作流规则，只输出数据和建议，决策权在人。

## 触发条件
用户使用 `/er-metrics`，或说"回顾指标""效率分析""看看耗时"。

## 数据来源
读取 `doc/{需求目录}/workflow/summary-*.md` 中的所有打点记录。

## 分析框架（三层）

### 第一层：数据事实（模型主导，高可信度）
各阶段耗时、返工率、AI 纠正次数、趋势——纯数据，不加判断。

### 第二层：环节合理性审视（模型提供素材，人做判断）
对每个环节问三个问题：
1. 它拦截了什么？（从数据中找证据）
2. 它的成本是什么？（耗时 + 对开发节奏的打断）
3. 有没有更低成本的方式达到同样效果？
每条标注 [数据充分/数据不足/纯推测]，人决定是否采纳。

### 第三层：设计哲学检验（模型提供检验结果，人做反思）
用数据检验三条设计哲学是否在实践中成立：
- "文档是契约" → spec 被绕过的次数？代码偏离 spec 但没被拦截的次数？
- "人做判断，机器做执行" → 人审核时发现了什么？审核是否流于形式？
- "对抗熵增" → 随功能数增加，AI 纠正次数趋势？规则数和执行可靠性的关系？

### 分析红线
1. 没有数据支撑的建议不出现
2. 不为改进而改进
3. 标注每条分析的置信度 [高/中/低]
4. "我不知道"比"我猜"更有价值
5. 数据不足时直接写"数据不足，建议继续观察 N 个功能后重新评估"

### 业界实践引用规则
- 仅在对某个环节产生具体质疑时，搜索相关业界实践作为参考
- 搜索词必须从数据发现的具体问题出发，禁止泛搜
- 引用必须标注来源 URL + 发布时间，无法验证的不引用
- 引用是参考，不是论据——"别人这么做"不构成"我们该这么做"的理由

## 三、设计哲学检验

> 用数据检验熵减三条设计哲学在实践中的表现。
> 不是质疑哲学本身的对错，而是检验"设计意图是否在执行层面兑现了"。

### 3.1 "文档是契约，不是注释"

**检验：spec 作为契约，执行得怎么样？**
- spec 被绕过的次数（代码改了但 spec 没同步）：{N} 次
  - 被"改动影响判断"拦截的：{M} 次
  - 漏网的：{N-M} 次
- 置信度：[高/中/低]

**数据说明了什么：**
- {解读}

### 3.2 "人做判断，机器做执行"

**检验：人的审核真的在发挥作用吗？**
- review 打回次数：{N} / {总次数}
- 打回原因分类：{列举}
- review 未发现但 verify 暴露的问题：{N} 次

**审核成本检验：**
- review 平均耗时：{X}h
- 是否存在"审核者需要跨文件查看"的情况：{是/否}
- 置信度：[高/中/低]

### 3.3 "对抗熵增"

**检验：工作流有没有在对抗熵增，还是自身也在熵增？**
- AI 纠正次数趋势：{上升/下降/平稳}
- 工作流规则总数变化：{从 X 条到 Y 条}
  - 规则增加但 AI 纠正没下降 → 加规则没用，问题在结构
  - 规则增加且 AI 纠正下降 → 规则有效
  - 规则增加且 AI 纠正上升 → 规则可能互相冲突
- 置信度：[高/中/低]

**本次是否发现新的熵增信号：**
- {列举，或"未发现"}

## 执行流程

1. 扫描 `doc/*/workflow/summary-*.md`，读取所有打点记录
2. 按功能分组，计算各阶段耗时（排除 >2h 间隔）
3. 按模板生成第一节（数据事实）
4. 基于数据生成第二节（环节审视），对有质疑的环节搜索业界实践
5. 基于数据生成第三节（哲学检验）
6. 汇总行动建议
7. 输出报告文件，同时在会话中展示摘要
