# Code Review Skill

> ⚠️ **Git 操作约束**：本阶段禁止一切 git 操作。Code review 只读代码，不修改、不提交。

## 效能打点
触发时追加 `- [{时间}] review | 开始审核 {审核范围概述}` 到对应功能的 summary 文件（如能确定功能；跨功能审核则追加到所有相关功能的 summary）。

## 触发后：一次问清楚

合并为一条消息问用户：

1. **审核范围**：哪些目录或文件？
2. **设计文档位置**：设计文档在哪个目录？（如果有的话）
3. **审核维度**：
   - A. 功能设计审核（对照设计文档，找实现与设计的偏差）
   - B. 纯代码审核（不依赖设计文档，审代码本身的质量问题）
   - C. 两者都要
4. **报告输出目录**：审核结果写到哪里？（默认 `doc/`）

等用户回复后再继续。

---

## 第一步：探索代码结构

列出审核范围内所有代码文件，推断项目的分层方式（不预设层级名称）：

```bash
# 示例：列出审核范围内的代码文件，按实际语言调整
find <审核范围> -type f \( -name "*.go" -o -name "*.py" -o -name "*.ts" -o -name "*.java" \) | sort
```

根据目录结构判断分层（如 handler/controller/service/repo/dao/model 等，具体取决于项目技术栈），理解各层职责后再开始审核。

---

## 第二步：读取并提炼设计文档（维度 A/C 时）

逐一读取设计文档，**只提炼文档中实际存在的内容**，不强求文档覆盖所有维度：

- 接口定义（入参、出参、字段名）→ 有则提炼
- 业务流程（关键步骤、分支条件、业务规则）→ 有则提炼
- 数据表结构（字段名、字段含义、枚举值说明、索引）→ 有则提炼
- 缓存设计（key 格式、过期时间）→ 有则提炼
- 其他约束（幂等要求、并发说明、降级策略等）→ 有则提炼

提炼完成后，整理出每个功能"设计文档要求了什么"，这是功能设计审核的唯一锚点。

---

## 第三步：生成审核规划

根据功能数量和代码体量，将任务拆成若干**批次**，每批 1-3 个功能模块。

规划格式：

```
## 审核规划

- 维度：XXX
- 输出文件：
  - 功能设计审核 → <用户指定目录>/code_review_by_design_doc.md
  - 纯代码审核   → <用户指定目录>/code_review_by_code.md

| 批次 | 覆盖模块 | 涉及文件 | 状态 |
|------|---------|---------|------|
| 1    | 模块A、模块B | ... | 待执行 |
| 2    | 模块C | ... | 待执行 |
```

**输出规划给用户确认一次**，确认后自动执行，不再打断。

---

## 第四步：按批次执行审核

每批次：
1. 全量读取该批次涉及的代码文件
2. 按维度审核（见下方审核方法）
3. 将结果**追加写入**对应报告文件（每次写入不超过 80 行，用 `cat >>` heredoc 方式）
4. 直接开始下一批次

遇到以下情况才主动暂停：
- 文件读不到、目录结构与预期不符
- 需要用户澄清的歧义（代码与文档均不明确）

---

## 审核方法

### 维度 A：功能设计审核

**核心方法：以设计文档为锚点，找"应该有但没有"和"有但做错了"**

不套固定清单，而是先问：
> 设计文档描述了哪些要求？代码是否一一对应实现了？实现的值/逻辑/字段是否与文档一致？

在此基础上，以下是常见的偏差模式（辅助参考，不限于此）：

- **枚举值/状态值与文档说明不一致**：文档说字段含义是 `1=子账号 3=主账号`，代码常量是否对应
- **流程步骤缺失**：文档描述了某个校验/过滤步骤，代码中是否存在
- **字段未传入**：文档要求落库的字段，代码中是否始终为空
- **缓存 key/TTL 与文档不一致**：key 格式、前缀、过期时间是否按文档定义
- **索引依赖未建立**：代码依赖某唯一索引做幂等，但文档/DDL 中没有定义该索引
- **业务规则未实现**：文档描述的特定条件（时间窗口、状态限制等），代码是否遗漏

> 文档没有描述的内容，不在功能设计审核范围内，归入纯代码审核。

---

### 维度 B：纯代码审核

不依赖设计文档，从代码本身发现问题：

**性能**
- 循环内是否有单条查询数据库/缓存/RPC（N+1）
- 可批量化但未批量化的操作

**正确性**
- 并发/异步任务是否正确处理了上下文生命周期（如请求取消后异步写入是否受影响）
- 数值/时间单位假设是否一致（如某处按秒处理，另一处按毫秒）
- 空切片/nil 入参是否有防御

**错误处理**
- 同层（如所有 handler 或所有 service 方法）错误处理策略是否统一
- 业务错误通过 code/msg 返回还是 error 返回，是否混用
- 错误日志是否包含足够上下文（ID、关键参数）

**代码质量**
- 定义但从未调用的方法/接口
- 未使用的 import（部分语言编译报错）
- 相同逻辑在多处分散维护
- 未使用的变量、常量

**幂等性**
- 写入操作是否有重复提交保护（唯一约束 + 冲突处理）

> 以上是常见模式，不限于此。审核时根据代码实际情况判断，不强求每项都有发现。

## 报告格式

### `code_review_by_design_doc.md`（维度 A）

```markdown
# Code Review - 功能设计审核

> 审核时间：YYYY-MM-DD
> 审核范围：xxx
> 设计文档：xxx

---

## 一、模块名（接口路径）

### 对应文件
- ...

### ✅ 符合设计的部分
- ...

### ⚠️ 问题

**【P1】问题标题**
问题描述...
建议：...

---

## N、跨模块问题（如有）

---

## 汇总

| 编号 | 模块 | 严重程度 | 问题描述 |
|------|------|----------|----------|
```

### `code_review_by_code.md`（维度 B）

```markdown
# Code Review - 纯代码审核

> 审核时间：YYYY-MM-DD
> 审核范围：xxx

---

## 一、模块名

### ⚠️ 问题

**【类别-P1】问题标题**
问题描述...
建议：...

---

## N、跨模块问题（如有）

---

## 汇总

| 编号 | 模块 | 类别 | 严重程度 | 问题描述 |
|------|------|------|----------|----------|
```

### 严重程度定义
- 🔴 高：影响功能正确性、数据写错、编译/运行时错误
- 🟡 中：逻辑隐患、性能问题、错误处理不当
- 🟢 低：代码质量、可维护性、文档一致性问题

## 审核完成打点
报告生成完成后，追加 `- [{时间}] review | 审核完成 (🔴{N} 🟡{M} 🟢{K})` 到对应功能的 summary 文件。
