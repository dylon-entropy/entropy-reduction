# 按功能设计实现代码

> ⚠️ **Git 操作（强制禁止）**：除非用户在当前消息中明确写出"提交"或"commit"，否则绝对不执行任何 git 操作（add / commit / push / stash / status）。违反视为严重错误。

## 触发条件
用户提供 spec 文件并要求实现代码，或使用 `/er-dev` 指令。

## 前置准备

### 0. 更新工作流进度 + 效能打点（第一步执行）
**效能打点**：追加 `- [{时间}] dev | 开始实现 {功能名}` 到 `workflow/summary-{功能编号}-{功能名}.md`。
将 `{需求目录}/workflow/progress.md` 更新为当前功能：
```markdown
需求目录: {需求目录名}
当前阶段: er-dev
当前功能: {当前 spec 对应的功能名}
状态: active
更新时间: {ISO 8601}
```
如 workflow/ 目录或 progress.md 不存在，创建之。

### 必读文件
1. **spec 文件**（自包含：已内联全局约束相关片段 + 现状），包括末尾「人工确认区」的补充信息
2. **`{项目根目录}/doc/pitfalls.md`**（如存在），所有踩坑记录作为硬性约束
3. **同目录下 `red-lines.md`**，所有条目为硬性约束

### 前置分析
4. 如项目有编码规范 skill，读取并理解分层架构和各层职责
5. 读取所有要调用的第三方包/模块的类型定义文件，确认字段名和类型，**禁止猜测**
6. 查看现有代码中同层的文件组织模式，新文件遵循相同模式
7. 读取项目数据层已有实现，理解封装模式

### 前置 Git 检测
8. **调用 er-git 工具（时机A）**：检测当前分支是否正确，不对则切换（此时无改动，切换无成本）

### 可复用代码扫描
前置准备完成后，列出本功能可复用的已有方法：
```
📦 可复用的已有代码：
- [包名.方法名]：[用途 + 本功能哪个环节用到]
（无则输出：未发现可复用的已有封装）
```

## 实现要求

### 1. 严格按 spec 实现（spec 只读）
- 实现过程中**不得自行修改 spec 文件**
- 数据库、接口、参数校验、流程、关键决策——全部按 spec
- 发现设计问题：先指出 → 说明原因 → 给建议 → **不要默默修正**

### 2. 遇到不确定时（强制停止）
spec 未覆盖的细节、文档歧义、外部系统行为不确定——**立即停止，列出所有不确定点，等用户回复**。禁止自行假设推进。

### 3. AI 代码质量红线
读取 `red-lines.md`：禁止循环操作外部存储、禁止伪造不确定值、禁止跳过已有抽象层、禁止重复实现已有功能、禁止 er-dev 阶段执行任何 git 命令。

### 4. 分层职责（强制）
先看项目已有代码的分层方式，新功能保持一致。有编码规范 skill 则遵守。

### 5. 通用规范对照（强制）
spec 文件中内联的通用规范条目，**在每个数据操作实现时逐条检查**。数量限制类约束不仅约束入参，也约束内部的批量查询和第三方调用。

### 6. 项目规范
遵守项目现有代码规范，使用现有工具库，保持风格一致。

### 7. 编译验证后（强制）
编译通过后**禁止顺手 git 操作**。继续后续步骤。

### 8. 输出纪律
只输出代码变更和必要说明。不生成总结文档、变更清单等衍生物。遗留问题用 TODO 注释。

### 9. 产物管理
非代码文件（建表 SQL 等）放在需求目录下按类型分子目录（如 `{需求目录}/sql/`），文件名含功能归属。

### 实现完成后 checklist（必做，按顺序）

1. **文档对齐**：对比 spec vs 实际实现，有偏差则以实际实现为准更新 spec（人工确认区内容融合进正文），然后调 er-sync（时机3）。这是实现阶段唯一允许修改 spec 文件的时机。
2. **自检**：复杂功能（并发/外部调用/改动超3文件/spec有A~E命中项）自动触发 er-review，审核结论合并到结束输出；所有功能对照可复用代码清单检查重复实现。
3. **环节结束输出**（禁止在用户审核前进入任何后续步骤）：
```
---
✅ {功能名} 实现完成，等待你审核

产物文件：
- {路径}（说明）

需要你做：
- {具体操作，如"执行建表SQL"，或"无"}

⚠️ 重点审核（最多3条，没有则省略）：
- {文件路径}：{一句话说明}

【自动 Review 结论】（复杂功能时输出）
- 🔴/🟡/✅ ...

审核完成后：
- **"验证"** — 开始测试
- **"review"** — 补做完整 er-review
---
```
4. **静默操作**（不输出，但必须执行）：更新 progress.md 当前功能 + 效能打点（追加 `- [{时间}] dev | 实现完成，等待审核 (AI纠正: {N}次)` 到 summary 文件，N 为人手动纠正次数）+ 学习信号检测（用户做了实质性代码修改/发现 spec 设计问题/指出 AI 错误模式时调 er-learning，固化后新增条目到 red-lines.md）
