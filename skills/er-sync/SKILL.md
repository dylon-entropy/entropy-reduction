# 文档一致性检测

> 按触发时机执行对应的同步流程，每个时机的输入输出明确，纯机械操作。

## 时机1：全局约束变更后

**触发**：全局约束.md 被修改后调用。
**输入**：全局约束.md（变更后）
**操作**：
1. 读取 `specs/` 目录下所有 spec 文件
2. 找出内联了全局约束内容的 spec（搜索受变更影响的数据格式约定、通用规范、技术栈）
3. 列出需要更新的 spec 及具体位置
4. **自动可修复的**（格式约定变更等）：直接更新，告知用户改了什么
5. **需要人决策的**（影响设计逻辑）：列出标 ❓，等用户确认
6. 更新进度文件（workflow/handoff.md）中的功能清单（如功能清单有变化）

## 时机2：新增/修改 spec 后

**触发**：新增 spec 文件、或 spec 内容发生修改（包括 er-dev 中人确认后的 spec 修改）。
**输入**：变更的 spec 文件路径
**操作**：
1. 读取全局约束.md 功能清单
2. 检查该 spec 对应的功能是否在功能清单中
   - 不在 → 追加到功能清单（序号、名称、描述、依赖），告知用户
   - 已在 → 检查功能名/描述是否需要更新
3. 检查 workflow/handoff.md（如存在）进度表是否包含该功能
   - 不在 → 追加为 ⏳（待开始）
4. 如果 spec 修改影响了其他 spec 引用的内容（表结构、接口定义）：
   - 列出受影响的 spec 及具体引用位置
   - 自动可修复的直接改，需人决策的标 ❓

## 时机3：code review / 文档对齐后

**触发**：er-dev 文档对齐完成后调用，或 er-review 发现实现与 spec 偏差后调用。
**输入**：被修改的 spec 文件路径
**操作**：
1. spec 已由调用方更新完毕（反向同步实现偏差）
2. 检查 spec 变更是否影响全局约束（新增数据格式、新外部系统、技术决策变更）
   - 有影响 → 更新全局约束对应部分，告知用户
   - 无影响 → 静默
3. 检查 spec 变更是否影响其他 spec 的引用（同时机2第4步）

## 输出格式

```
【一致性检测结果】时机{N}
✅ 自动同步：
- {文件路径}：{修改内容}

❓ 需要确认：
- {文件路径}：{问题描述}

（全部一致时输出：✅ 无需同步）
```
