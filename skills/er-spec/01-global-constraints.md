# 阶段1：生成全局约束

> 本文件由 `SKILL.md` 调度加载，不要单独触发。

## 前置依赖
- 用户提供的需求文档、会议记录等输入文件
- 项目现有代码（如用户提供或 @ 引用）

## 执行
0. **效能打点**：创建 `doc/{需求目录}/workflow/summary-{首个功能编号}-{功能名}.md`（如功能清单尚未生成则在步骤4后补建），追加 `- [{时间}] spec | 开始生成全局约束`
1. 阅读所有输入文件 + 项目现有代码，提取关键信息
2. 确定需求名称（简短，用于目录命名），无法推断时询问
3. 确定需求目录：
   - 输入文件已在 `doc/YYYYMMDD-*/` 目录下 → 使用该目录
   - 否则 → 创建 `doc/YYYYMMDD-需求名称/`，将输入文件复制到该目录
4. 生成 `全局约束.md`（含功能清单）

## 全局约束模板

```markdown
# 全局约束

> 由AI基于需求文档和会议记录生成，请审核并补充。
> ❓ 标注表示AI不确定，请重点关注。

## 改动类型
[新功能 / 改已有功能]

## 系统边界

### 我们的系统
- [系统名称和职责]

### 外部系统
| 系统 | 交互方式 | 数据流向 | 说明 |

### 明确不做的事
- [从输入文件中提取]

## 数据格式约定
| 项目 | 格式 | 示例 |

## 技术栈
| 类别 | 选型 | 说明 |

## 测试执行

> ⚠️ **必填项**：后续验证阶段的唯一依据。
> **所有 ❓ 必须在阶段1完成前清零。**

| 项目 | 值 |
|------|-----|
| 测试目录 | |
| 运行命令 | |
| 配置文件 | |

## 团队约定

## 通用规范
[跨功能约束，如批量上限、超时策略等。没有则删除此节]

## Git 分支
| 项目 | 值 |
|------|-----|
| 分支名 | feature/YYYYMMDD-{关键词} |

## 功能清单
| 序号 | 功能名称 | 一句话描述 | 依赖 |

---
## 人工确认区

> 填完后说"继续"。

### 待确认问题
### 测试执行（必填，不允许带❓进入阶段2）
### 关联已有代码（改已有功能时填写）
### 补充信息（可选）
```

## 生成要求
- 不确定的地方**双重标注**：原位 ❓ + 汇总到"待确认事项"
- 从项目代码推断技术栈和数据格式
- 只写已识别到的内容，不编造
- 功能清单只列输入文件中明确提到的功能

## 输出后
告知：`全局约束.md 已生成，请查看末尾「人工确认区」填写后说"继续"。`

用户说"继续"后：
1. 读取「人工确认区」所有已填内容
2. 未填的 ❓ 一次性列出，等补充
3. **专项检查「测试执行」**：三项全无 ❓ 才能进阶段2
4. 如有「关联已有代码」关键词：搜索匹配 → 唯一则回写路径，多个则让用户选择
5. 所有项确认完毕后，调用 **er-git 工具创建分支**
6. 创建 `{需求目录}/workflow/progress.md`：
   ```markdown
   需求目录: {需求目录名}
   当前阶段: er-spec
   当前功能:
   状态: active
   更新时间: {ISO 8601}
   ```
7. **效能打点**：追加 `- [{时间}] spec | 全局约束确认，分支已创建，进入功能设计`
8. 告知：`全局约束已就绪，分支已创建，进入功能设计。`

**停止，等待用户说"继续"后才进入阶段2。**
