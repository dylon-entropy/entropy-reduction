# 阶段2：按依赖顺序逐个生成自包含 Spec

> 本文件由 `SKILL.md` 调度加载，不要单独触发。
> ⚠️ **Git 操作约束**：本阶段禁止一切 git 操作。分支已在阶段1创建完成。

## 前置依赖
- **必读**：`全局约束.md`（改动类型、数据格式约定、功能清单、关联已有代码、补充信息）
- **必读**：需求文档 + 会议记录（原始输入）
- **必读**：`{项目根目录}/doc/pitfalls.md`（如存在），所有踩坑记录作为设计约束
- 已生成的前置 spec 文件（按依赖顺序读取）

## 核心原则

### Spec 文件必须自包含
每个 spec 文件是 er-dev 阶段的**唯一设计输入**。er-dev 不再读全局约束，所以 spec 必须包含实现所需的全部信息：
- 全局约束中与本功能相关的数据格式约定、通用规范条目——内联到 spec 对应位置
- 改已有功能时，项目中要改的现有代码的现状——内联到 spec 的"现状"section
- 其他 spec 中本功能依赖的表结构/接口定义——内联引用

### 文件是状态，对话不是
每个功能生成完成并**写入文件**后，后续功能必须**从文件读取**前置功能的表结构和接口定义。

### 只输出实际涉及的 section
不生成空 section 占位。

## 执行
1. 创建 `specs/` 目录
2. 检查 `全局约束.md` 中的"改动类型"，读取「人工确认区」中的补充信息和关联已有代码
3. **分析功能间依赖关系**（阶段2开始时执行一次）：
   - 逐个功能分析：它依赖哪些**表结构**和**接口**
   - 输出依赖顺序，格式：`功能01 → 功能03 → 功能02`，等用户确认后开始生成
4. **按确认的依赖顺序逐个生成**：
   - **效能打点**：创建/追加 `doc/{需求目录}/workflow/summary-{功能编号}-{功能名}.md`，写入 `- [{时间}] spec | 开始设计 {功能名}`
   - 判断功能复杂度（见下方）
   - 生成自包含的 spec 文件
   - 复杂功能：设计自审（见下方），发现问题则追问
   - 开始下一个前，**从文件读取**已生成的前置 spec
5. 所有功能生成完成后，执行跨 spec 交叉验证（见下方）
6. 等待用户逐个审核

## 复杂度分级

**简单功能**（满足全部）：单接口 CRUD、无外部系统调用、无并发、无异步
- 直接生成，跳过方案对比和设计自审
- 无 A~E 命中项时不打断，直接继续下一个

**复杂功能**（满足任一）：涉及异步/Job/MQ、外部系统调用、并发/分布式、数据量大、多步骤流程
- 完整生成：方案对比 + 设计自审 + A~E 追问
- 生成后展示自审结果和追问，等用户回复

## Spec 自包含——内联规则

生成每个 spec 时，从全局约束中提取**与本功能相关的**内容内联：

1. **数据格式约定**：本功能涉及的 ID 格式、时间格式、响应格式 → 内联到接口 section
2. **通用规范条目**：本功能涉及的约束（如批量上限、超时策略）→ 内联到相关 section
3. **技术栈选型**：本功能用到的组件（如 Redis、MQ）→ 内联到对应设计 section

**不内联的**：与本功能无关的全局约束内容（其他功能的格式约定、不涉及的技术栈等）。

内联格式：在 spec 文件相关 section 中自然融入，不单独开"全局约束摘要"section。

## 改已有功能时的现状 section

当"改动类型"为"改已有功能"时，生成每个 spec 前：
1. 读取功能清单中该功能的"关联已有代码"字段
2. 读取对应文件，提取与本功能相关的：表结构、接口定义、核心业务逻辑
3. 在 spec 文件头部加"现状"section

```markdown
## 现状

> 从现有代码中提取，仅列出与本次改动相关的部分。

### 现有表结构（如有改动）
[相关表的当前字段，只列改动涉及的表]

### 现有接口（如有改动）
[相关接口的路径、关键参数、响应结构]

### 现有业务逻辑
[核心流程简述，只写与本次改动有交集的逻辑]

### 本次改动范围
- 新增：[新增的内容]
- 修改：[修改的内容]
- 不变：[明确保持不变的内容]
```

## 可选 Section 菜单

根据功能实际涉及的内容选择输出，不涉及的不生成：

| Section | 何时输出 |
|---------|---------|
| 现状 | 改已有功能时 |
| 流程 | 有多步骤业务流程时 |
| 关键决策 | 有非平凡技术决策时 |
| 数据库设计 | 有新建表/改表时，给完整 CREATE/ALTER TABLE SQL |
| 缓存设计 | 有缓存时，含 key 格式、过期时间 |
| 接口 | 有新增/修改接口时，含完整 JSON 示例 + 参数校验表格 |
| 外部依赖 | 调用外部系统/第三方 API 时 |
| 异步流程 | 有 Job/MQ/事件驱动时 |
| 人工确认区 | **始终输出** |

## 方案对比（复杂功能时触发）

存在**非平凡技术决策**时（多种实现路径且各有代价），在"关键决策"section 中输出：

```markdown
### 决策：[决策描述]
| | 方案A：[名称] | 方案B：[名称] |
|---|---|---|
| 优点 | | |
| 缺点 | | |
| 适用场景 | | |

**推荐**：方案X，理由：[一句话]
```

纯 CRUD、逻辑唯一时：直接写结论，不输出对比表格。

## 人工确认区模板

```markdown
---
## 人工确认区

> 填完后说"继续"。

### 回归测试范围
（AI 基于改动推断；确定无影响时此节省略）
- `METHOD /path/to/api` — [受影响原因]：需要 / 不需要

### 📋 缺少外部信息
### ⚠️ 高风险项
### ❓ AI 不确定
### 📌 其他
### 补充信息
```

## 设计自审（复杂功能生成后执行）

生成复杂功能的设计后，从以下角度审视，发现问题则追问：

1. **数据流完整性**：数据链路上有没有断点或未定义的环节？
2. **失败场景**：每个外部调用失败时怎么处理？
3. **规模适配**：数据量级是否影响方案选择？
4. **安全**：SQL 注入风险？权限校验完整？敏感数据泄露？
5. **向后兼容**：老客户端兼容？枚举扩展？字段类型预留？
6. **A~E 高频场景**：是否命中后台任务/消息消费/分页翻库/多实例并发/灰度控制？

能自行修正的直接改文件并说明；需要用户决策的追问等回复。

## A~E 高频场景参考知识

### A：后台定时任务 / Job
追问：一次性 vs 持续运行、断点恢复策略、幂等性保证

### B：消息队列消费
追问：过滤规则、幂等性保证

### C：分页翻库
追问：排序字段、大量同值数据的翻页策略

### D：多实例并发
追问：并发控制策略（分布式锁/分片/幂等）

### E：灰度发布控制
追问：控制维度、默认状态

## 跨 Spec 交叉验证（所有功能生成完成后自动执行）

所有 spec 生成完成后，**在用户审核前**自动检查：
1. **表结构一致性**：同一张表在多个 spec 中的字段定义是否一致
2. **接口依赖一致性**：spec A 调用 spec B 的接口时，入参/出参是否对得上
3. **枚举值一致性**：多个 spec 引用同一状态字段时，定义是否一致
4. **缓存 key 冲突**：不同 spec 的缓存 key 是否有冲突

发现不一致：输出报告，自动修复可修复的，需人决策的标 ❓。全部一致则静默。

## 生成要求
- 只生成本功能相关的内容
- 不确定的点放在"待确认"中
- 接口给出完整 JSON 示例
- **数据库设计给出完整可执行的 CREATE TABLE SQL**（含索引）
- 接口增加参数校验表格
- 多个功能之间的表结构、缓存设计保持一致
- 回归测试范围每条写明接口路径或函数名

## 用户审核修改后的处理
用户修改 spec 文件时：
1. 应用用户的修改
2. 判断修改是否影响全局约束或其他 spec
3. 如有影响，调用一致性工具同步

## 输出后
**效能打点**：为每个已生成的功能追加 `- [{时间}] spec | spec 设计完成，等待审核`。
告知用户：所有 spec 已生成，请逐个审核 `specs/` 下的文件，全部确认后说"继续"。

**停止，等待用户说"继续"后才进入阶段3。**
