# 生成技术文档

> ⚠️ **Git 操作约束**：本阶段唯一的 git 操作是阶段1确认后调用 er-git 工具创建分支。除此之外禁止任何 git 操作。

## 触发条件
用户提供会议记录、需求描述或相关文件，要求生成技术文档。

## 阶段判断与调度

根据需求目录的当前状态判断从哪个阶段开始，**读取对应的阶段文件**执行。

### 判断流程

1. `全局约束.md` 不存在 → **阶段1**，读取 `01-global-constraints.md`
2. `全局约束.md` 存在 → 执行一致性检查（见下方），然后：
   - 功能清单中有未设计的功能 → **阶段2**，读取 `02-spec-design.md`
   - 所有功能已设计，`技术文档.md` 不存在 → **阶段3**，读取 `03-doc-assembly.md`

### 一致性检查（阶段判断时自动执行）

当 `全局约束.md` 存在时，在判断阶段之前：
1. 读取功能清单 + `specs/` 文件列表
2. specs/ 有但功能清单没有 → 提示用户确认后追加
3. 判断 spec 是否已设计：内容含技术 section（流程/接口/数据库设计/缓存设计）则已设计

> 一致性检查静默执行，只在发现差异时输出。

### 执行流程
1. 判断当前阶段
2. 读取对应阶段文件执行
3. 阶段完成后**停止**，等用户说"继续"
4. 每次只读当前阶段文件

## 阶段间数据依赖

通过**产出文件**传递，不通过对话记忆：
- 阶段2 读取阶段1 的 `全局约束.md`
- 阶段3 读取 `全局约束.md` + `specs/*.md`

## 全流程注意事项

1. 不推测补充功能：只生成输入文件中明确提到的
2. 全局约束是活文档：整个流程中持续引用和更新
3. 每个阶段等待用户确认
4. 阶段间不自动跳转
