# 验证与测试

> ⚠️ **Git 操作约束**：测试全部通过后调用 er-git 工具（时机B）提交。除此之外禁止任何 git 操作。

## 触发条件
用户确认代码实现完成，要求验证，或使用 `/er-test` 指令。

## 效能打点
触发时追加 `- [{时间}] verify | 开始验证 {功能名}` 到对应功能的 `workflow/summary-{功能编号}-{功能名}.md`。

## 前置准备
- 读取 spec 文件，提取所有需要验证的行为
- **读取 spec 中"回归测试范围"**（如有）：用接口路径在项目中搜索对应文件，纳入测试范围
- 读取实现代码，了解结构
- 读取 `全局约束.md` 获取测试配置（测试目录、运行命令、配置文件）
- 读取项目现有测试代码，学习模式

## 测试运行命令（唯一来源）

从 `全局约束.md`「测试执行」章节读取。如果没有或未填写，**立即停止**：
```
【缺少测试配置】
全局约束.md「测试执行」章节未填写。请补充后再说"验证"。
```

## 测试设计原则

1. **真实环境，不 mock**
2. **少量精准数据**：每个场景最少数据量，使用可识别的测试值（如 user_id=99999001）
3. **不自动清理**：测试数据保留，方便观测
4. **覆盖范围**：正常流程 + 参数校验 + 业务异常 + 回归范围（每项至少1个 happy path）

## 执行流程

### 1. 生成测试代码
- 在项目已有测试目录下创建（遵循已有组织方式）
- 测试函数命名：`Test_{接口名}_{场景描述}`
- 如需更新 TestMain（新增全局变量和初始化），一并完成

### 2. 自动运行
- 使用全局约束中声明的运行命令
- `-run` 指定本次测试函数（避免运行无关测试）
- 生成后直接运行，不等待确认

### 3. 结果判断

**全部通过** → **效能打点**：追加 `- [{时间}] verify | 测试通过 (自动修复: {N}轮)` 到对应 summary 文件。然后调用 **er-git 工具（时机B）** commit + push，输出摘要：
- 通过的测试数量和列表
- 创建的测试数据汇总
- Git 提交信息

摘要末尾：`测试通过，代码已提交并推送。等待你确认下一步。`

测试全部通过后，静默更新 `workflow/progress.md`：当前阶段改为 `completed`。

**有失败** → 分析错误类型：

| 错误类型 | 处理 |
|---------|------|
| 编译错误 | 自动修复，重新运行 |
| 断言失败 | 分析后自动修复，重新运行 |
| 空指针/越界 | 自动修复，重新运行 |
| DB/Redis 连接失败 | **停止**，报告 |
| 表不存在 | **停止**，报告 |
| 框架初始化失败 | **停止**，报告 |
| 超过3轮修复仍失败 | **停止**，报告完整错误和已尝试的修复 |

### 4. 停止报告格式
```
【测试中断】
- 错误类型：[基础设施/外部服务/框架配置/超过修复上限]
- 错误信息：[完整错误]
- 需要你做：[操作指引]
- 修复后说"继续"
```

## 输出要求
- 测试代码是项目永久资产
- 测试函数有场景描述注释
- 关键测试数据用常量定义
