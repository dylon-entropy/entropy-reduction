# 快速上手示例

## 场景

你要给一个 Web 应用添加"用户标签管理"功能：
- 管理员可以创建、编辑、删除标签
- 可以给用户批量打标签
- 支持按标签筛选用户

## 第一步：初始化

> 指令方式：`/er-init`
> 自然语言：说"熵减初始化"

```
你：/er-init

AI：熵减工作流初始化完成。
✅ 工作流优先级已声明
✅ 项目目录已就绪
📋 踩坑记录：0条

未检测到交接文件，发送 /er-spec 或"生成技术文档"开始。
```

## 第二步：生成技术文档

> 指令方式：`/er-spec`
> 自然语言：说"生成技术文档"或"写 spec"

```
你：/er-spec
    需求：用户标签管理功能，包括标签 CRUD、批量打标签、按标签筛选用户。

AI：（生成全局约束.md，包含功能清单、技术栈、数据格式约定等）
    全局约束.md 已生成，请查看末尾「人工确认区」填写后说"继续"。
```

你审核全局约束，填写测试配置等必填项，然后说"继续"。

```
AI：（分析功能依赖，按顺序生成每个功能的自包含 spec）
    所有 spec 已生成，请逐个审核 specs/ 下的文件。
```

此时 `doc/` 目录结构：

```
doc/20260404-用户标签管理/
├── 需求描述.md
├── 全局约束.md
├── 技术文档.md
├── specs/
│   ├── 01-标签CRUD.md
│   ├── 02-批量打标签.md
│   └── 03-按标签筛选用户.md
└── workflow/
    └── progress.md
```

## 第三步：按 spec 实现代码

> 指令方式：`/er-dev`
> 自然语言：说"按设计实现"或"实现这个功能"

```
你：/er-dev

AI：（读取 spec，扫描可复用代码，开始实现）
    ...
    ✅ 标签CRUD 实现完成，等待你审核
    
    产物文件：
    - 接口层：标签相关 API 处理
    - 业务层：标签业务逻辑
    - 数据层：标签数据存取
    
    审核完成后：
    - "验证" 或 /er-test — 开始测试
    - "review" 或 /er-review — 补做完整代码审核
```

## 第四步：验证测试

> 指令方式：`/er-test`
> 自然语言：说"验证"或"跑测试"

```
你：/er-test

AI：（自动生成测试代码，运行测试）
    ✅ 测试通过（6/6）
    代码已提交到 user-tags 分支
```

## 第五步：继续下一个功能或交接

```
你：继续下一个            # 自然语言，AI 自动识别
    或 /er-dev           # 指令，直接进入下一个功能的实现

你：/er-handoff          # 如果要换会话
AI：交接文件已生成。新建会话发送 /er-init 或"熵减初始化"即可恢复。
```

## 日常改动（不走完整流程）

已有功能需要改动时，不用走完整的 spec → dev → test 流程。直接对话：

```
你：帮我在标签接口加一个 description 字段

AI：（常驻规则自动触发改动影响判断）
    检测到 spec 文件 specs/01-标签CRUD.md 中定义了该接口。
    影响范围：
    - specs/01-标签CRUD.md：接口定义、参数校验表
    - 数据层：表结构需加字段
    
    建议先改 spec 再改代码，确认吗？
```

> 这就是"对话即工作流"——你不需要记指令，常驻规则会自动判断该走什么流程。

## 效能参考

来自真实项目效能打点数据（简单功能，单接口）：

| 阶段 | 耗时 |
|------|------|
| spec | ~2 min |
| dev | ~4 min |
| verify | ~4 min（外部依赖就绪时） |
| **端到端** | **~10 min** |

中等复杂度功能（多接口、有缓存/外部系统）预计 20-40 分钟。

> 熵减内置效能打点（er-metrics），你可以用自己的数据来验证。
