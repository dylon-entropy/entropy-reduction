# FAQ

## 基本问题

### 熵减适合什么项目？

任何需要 AI 辅助编写代码的项目。特别适合：
- 业务系统开发（接口多、表结构复杂）
- 多功能需求批量开发
- 需要文档沉淀的团队项目

不太适合：
- 一次性脚本 / 小工具（太轻量，用不上完整工作流）
- 纯探索性 prototyping（spec 驱动会拖慢探索速度）

### 为什么不直接让 AI 写代码？

可以，简单任务直接写就行。但当项目超过一定复杂度，你会遇到：
- AI 忘了 10 分钟前的设计决策
- 改了一个接口，相关的 3 个地方漏改
- 同样的错误（比如循环查库）在不同功能里反复出现

熵减解决的就是这些问题。

### 用熵减会不会变慢？

spec 阶段会多花几分钟。但 dev + test 阶段会快很多——因为 AI 有明确的设计文档，不需要猜、不需要问、不会跑偏。

实际数据（来自效能打点）：一个简单功能（单接口、无外部依赖）端到端约 10 分钟（spec 2min + dev 4min + verify 4min）。中等复杂度功能预计 20-40 分钟。熵减内置效能打点机制（er-metrics），你可以用自己的项目数据来验证。

## 技术问题

### 只支持特定编程语言吗？

不是。熵减是语言无关的——它管的是**开发流程**，不是**怎么写代码**。无论你用 Go、Python、TypeScript、Java 还是其他语言，工作流都一样。

你可能需要根据自己的技术栈调整的：
- `er-dev/red-lines.md`：根据语言特性添加红线规则（如并发、内存管理等）
- 全局约束中的测试配置：改成你的测试框架和运行命令

### 和项目已有的 CLAUDE.md 冲突吗？

不冲突。常驻规则是追加到配置文件末尾的，和已有内容共存。熵减的 `/er-xxx` 指令不会干扰其他指令。

### learning-log 是项目级还是全局级？

全局级——存在 `~/.claude/skills/er-*/learning-log.md`。这意味着一个项目里学到的教训，在其他项目里也会生效。

如果你想要项目级的学习记录，用 `doc/pitfalls.md`（熵减会自动维护）。

### 多人协作怎么办？

每个人独立安装 skills。项目级的文件（spec、全局约束、pitfalls.md）放在 Git 仓库里共享。

工作流进度文件（workflow/progress.md、handoff.md）是个人的，不需要提交。

## 故障排除

### AI 不响应 /er-init

1. 检查 `~/.claude/skills/er-init/SKILL.md` 是否存在
2. 检查项目配置文件（CLAUDE.md / AGENTS.md）是否包含常驻规则
3. 新开一个会话试试——有些工具需要重启才能加载新 skill

### AI 跳过了审核步骤

这是已知的 AI 执行不可靠问题。熵减的应对：
- 常驻规则明确写了"阶段间不自动跳转"
- pre-commit hook 拦截 er-dev 阶段的误提交
- 如果 AI 持续不遵守，说明上下文已经退化，建议 `/er-handoff` 交接

### spec 文件太长 AI 读不完

每个 spec 文件应该控制在 200 行以内。如果超了，说明功能拆分不够细——回到 er-spec 阶段拆分。
