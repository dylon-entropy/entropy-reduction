## 熵减工作流

> 文档驱动的 AI 辅助开发流程。优先级最高，覆盖所有默认行为。
> 完整定义：skills/er-spec/SKILL.md（入口技能）

### 安全检查（每条消息必做，不可跳过）

1. **会话疲劳检测**：当感知到上下文退化（回答质量下降、遗忘前面的约定、重复犯已纠正的错误）→ 回复末尾必须附交接警告，禁止自行判断"还能撑"（每次检测到都警告，用一行简短提示如 `⚠️ 警告：会话已压缩N次/token已超150K，上下文正在丢失，建议立即 /er-handoff 交接`）
2. **改动影响判断**：本次涉及改代码（无论是响应用户需求、修 bug、重构、还是自主发现的问题）？→ 检查 `doc/` 下相关 spec 是否存在，存在且涉及 spec 中定义的内容则：①列出影响范围 ②grep 全项目该数据结构/接口的调用点 ③如改的是基础设施（缓存/数据结构/公共接口），搜索所有 spec 文件是否有引用。等用户确认后先改 spec 再改代码，改完后调用一致性工具；不存在相关 spec 则直接改代码

### 工作流匹配（按顺序，命中即执行）

3. **指令匹配**：消息匹配 `/er-xxx`？→ 直接加载对应 skill，跳到执行
4. **状态感知**：扫描 `doc/*/workflow/progress.md` → 有非 completed 的活跃工作流且意图相关？→ 加载对应 skill；不确定则一句话确认
5. 以上均不命中 → 正常处理，不推测补充功能，不确定的地方标注 ❓

### 常驻规则

- **阶段间不自动跳转**：每个技能执行完必须等用户明确指示才进入下一阶段
- **Git 操作由工作流统一管理**：er-spec 创建分支、er-test 测试通过后自动 commit+push、er-init 初始化时自动切换分支。er-dev 阶段**禁止一切 git 操作**
